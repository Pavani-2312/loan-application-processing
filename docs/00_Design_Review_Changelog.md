# 00 · Design Review Changelog
## Project 05 — Loan / Credit Application Processing Agent

This log records an external design review of the v1.0 docs and exactly what changed in response. Read this first if you already know v1.0 — it tells you where to look for what's different. New readers can skip straight to `01_Requirements.md`.

| # | Finding | Verdict | Where fixed | What changed |
|---|---|---|---|---|
| 1 | Fairness check only tests extraction-layer identity leakage, not real disparate impact | Agreed — was overclaimed | `01_Requirements.md` §10 (new), FR-09; `02_Architecture.md` FairnessNode row; `03_Functional_Specification.md` §4; `05_UI_Design.md` §4 fairness panel | Renamed to "Identity-Blind Extraction Consistency Check" everywhere; explicit scope statement added; real disparate-impact analysis documented as an unsolved, out-of-scope limitation (L2), not silently implied to be covered. |
| 2 | No verifiable authentication on underwriter identity | Agreed — correctly flagged as blocking | `01_Requirements.md` §10 (new, L1); `05_UI_Design.md` §1, §7 | Elevated to a named Production Blocker with a persistent, undismissable UI notice. Not faked with a stub login — deliberately left visible and unsolved. |
| 3 | Chroma similarity search is the wrong mechanism for a regulatory citation | Agreed — redesigned, not patched | `02_Architecture.md` tech stack table, §7 decision log; `03_Functional_Specification.md` §2.4; `04_Data_Policy_Model.md` §2 | Citation is now a deterministic clause-ID lookup sourced from `policy_config.yaml` (exact-ID `get()` from Chroma, no similarity math). Chroma's semantic search is repurposed as an underwriter-facing manual policy search tool only — never on a path that produces decision evidence. A config/corpus ID mismatch is now its own error state (`POLICY_CONFIG_ERROR`), not a silent "no clause found." |
| 4 | Income stability has two dimensions (tenure, variability) but one undefined combined score — breaks determinism (NFR-02) | Agreed — real bug | `03_Functional_Specification.md` §2.2; `04_Data_Policy_Model.md` §1 (Section 5 split), §3 (`score_breakdowns.sub_factor`), §4 (config split into two band lists) | Defined explicitly: two independent sub-scores, each with its own policy clauses (5.1 tenure, 5.2 variability), combined via `min()` (weakest-link, conservative credit-risk convention). Fully deterministic and reproducible across implementations. |
| 5 | REFER as a human decision has no defined end state or routing | Agreed — real workflow gap | `01_Requirements.md` FR-11; `02_Architecture.md` HumanGateNode row; `03_Functional_Specification.md` §3.2; `04_Data_Policy_Model.md` §3 (`human_decisions` redesigned); `05_UI_Design.md` §4 decision form, §3 queue rows | Human REFER is now explicitly non-terminal: requires a categorical `refer_reason`, sets `REFERRED_FOR_ESCALATION` (not `DECIDED`), and re-queues the application. `human_decisions` now supports a full decision history per application; only an `APPROVE`/`DECLINE` row is terminal. |
| 6 | Schema-valid-but-hallucinated extraction (e.g. 36 months read from "3 months") is undetected | Agreed | `03_Functional_Specification.md` §1.2a (new), §8; `04_Data_Policy_Model.md` §3 (`extracted_fields` columns), §4 (`extraction_rules`); `05_UI_Design.md` §4 (Extracted Fields panel) | Every scoring-relevant extracted field now requires a confidence level and a literal evidence span from the source document. Low confidence blocks auto-scoring (`NEEDS_MANUAL_VERIFICATION`) until a human confirms/corrects the value in a new, always-visible Extracted Fields panel. |
| 7 | DTI (max-based) and credit history (min-based) band logic asymmetry was undocumented — an implementer could invert one | Agreed | `03_Functional_Specification.md` §2.2; `04_Data_Policy_Model.md` §4 | Each band list in `policy_config.yaml` now carries an explicit `direction: max_asc` / `direction: min_desc` field; the scoring engine branches on it rather than assuming. |
| 8 | Build prompt (doc 06) says "read all five docs" while six files exist total | Agreed, minor | `06_CLI_Build_Prompt.md` intro | Clarified: docs 01–05 are the five reference docs; 06 is the instruction wrapper, not a sixth reference doc. |
| 9 | Concurrency/idempotency under multi-session Streamlit + SQLite unaddressed | Agreed | `01_Requirements.md` §10 (L4); `02_Architecture.md` §6 (new subsection); `03_Functional_Specification.md` §8; `04_Data_Policy_Model.md` §3 (`applications.status_version`, `intake_idempotency_key`) | Added SQLite WAL mode, a single serialized writer path, optimistic locking on `applications.status`, and a client-generated idempotency key on intake. Documented as a real, only-partially-solved scaling limit — not claimed as production-grade concurrency handling. |

## What did *not* change

The core architecture the reviewer praised is untouched:
- LLM-for-understanding / Python-for-scoring split.
- Human gate as a database-level structural control, not a prompt instruction.
- Free text structurally excluded from the scoring path (guardrail design).
- Append-only audit model.
- Six-document, no-repetition doc structure (now seven, with this changelog as the index).

## Round 2 — second external review, findings and fixes

A follow-up review of the round-1 revision found the five biggest structural gaps well-fixed, and surfaced eight more issues — mostly specification gaps that would bite an implementer rather than architectural problems. All eight addressed below.

| # | Finding | Verdict | Where fixed | What changed |
|---|---|---|---|---|
| 1 | Concurrency mitigation listed a `threading.Lock`-based writer path alongside DB-level mechanisms — the lock gives no protection in a multi-process Streamlit deployment | Agreed | `02_Architecture.md` §6, decision log | Removed the in-process lock entirely. Concurrency safety now rests solely on SQLite WAL mode + optimistic locking on `applications.status_version`, which are correct regardless of process topology. |
| 2 | `NEEDS_MANUAL_VERIFICATION` correction "unblocks scoring" but the re-entry mechanism (what re-runs, what's preserved) was undefined | Agreed | `03_Functional_Specification.md` §1.2b (new); `04_Data_Policy_Model.md` §3 (`revision_number` added to `score_breakdowns`/`recommendations`/`fairness_checks`) | Correction resumes the graph from ScoringNode onward (not a full restart); each re-run produces new `revision_number`-tagged rows, nothing is overwritten, a `RESCORED_AFTER_VERIFICATION` audit event is logged. |
| 3 | DTI (`max`-based) bands and the "inclusive lower bound" description contradicted each other at exact boundary values | Agreed — genuine spec bug | `03_Functional_Specification.md` §2.2 | Removed the contradictory "inclusive lower bound" language; replaced with one precise rule per direction (`max_asc`/`min_desc`), with worked boundary examples, so a value exactly on a boundary has one unambiguous, testable resolution. |
| 4 | `extracted_fields` overwrote the original model-extracted value on human correction — no way to audit "did the system almost score with a different number" | Agreed | `04_Data_Policy_Model.md` §3 | `extracted_fields` is now append-only/versioned (`field_version`, `is_effective`) — corrections add a row, they never overwrite one. |
| 5 | Build prompt's two new test scenarios (REFER chain, low-confidence extraction) weren't in the requirements traceability table | Agreed | `01_Requirements.md` FR-16 (new), §8 traceability table | Added FR-16 for the extraction-confidence requirement; added both scenarios to the traceability table; REFER chain traced to existing FR-11/13/14. |
| 6 | "Fairness disparity rate" KPI is structurally ~0% by construction (numeric scorer can't see redacted fields) — conflates system correctness with actual fairness | Agreed | `01_Requirements.md` §5 | Renamed to "Identity-blind consistency pass rate" with an explicit framing note distinguishing it from a real fairness/disparate-impact metric (cross-referenced to L2). |
| 7 | `audit_log.event_payload` had no defined schema; no single-application, self-contained export existed (only bulk filtered CSV/JSON) | Agreed | `04_Data_Policy_Model.md` §3 (event_payload schema table); `05_UI_Design.md` §5 ("Generate Audit Package") | Defined required fields per `event_type`. Added a "Generate Audit Package" export producing one standalone, human-readable document per application — distinct from the bulk export. |
| 8 | No timeout, cancellation, or resumability defined for the multi-call agent pipeline | Agreed | `02_Architecture.md` §6 (new subsection); `05_UI_Design.md` §2 (spinner messaging) | Added a 30s per-node timeout, durable per-node writes (not just one write at the end), and pipeline resumability from the last completed node. No cancel button needed given resumability. |
